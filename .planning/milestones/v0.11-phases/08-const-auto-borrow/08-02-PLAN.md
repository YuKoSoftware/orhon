    **Step 4: Handle var callers of promoted functions** (Pitfall 2)

    Two approaches:
    (a) Extend MIR annotator to also annotate var callers when the target param is promoted — this would require a second pass or deferred annotation.
    (b) In codegen, when generating call arguments, check if the callee function has promoted params and emit `&` for non-primitive args at those positions even without a coercion annotation.

    Use approach (b) — it's simpler and keeps the change in codegen:

    Find where call arguments are generated in the MIR codegen path. Search for the call expression generation in `generateExprMir` or `generateCallMir`. When emitting each call argument:
    1. Check if this argument position is in `const_ref_params` for the callee function
    2. If the argument does NOT already have a `value_to_const_ref` coercion (i.e., it's a var caller)
    3. AND the argument is a non-primitive identifier or expression
    4. Then prepend `&` before the argument

    Look for the call arg generation code — it likely iterates over MirNode children for the call, using either `generateCoercedExprMir` or `generateExprMir`. The existing coercion path handles const callers. Add the var caller `&` emission for promoted params.

    **Step 5: Function body access for promoted params**

    When a param is promoted to `*const T`, code inside the function body that accesses the param might need `param.*` (dereference). However, Zig auto-derefs field access: `param.x` works for both `T` and `*const T`. The only case that breaks is bare param usage like `return param` or `var copy = param` — these would return/copy a pointer instead of a value.

    For this phase, rely on Zig auto-deref (which handles field access and method calls). If tests reveal cases needing explicit deref, handle them incrementally. The research confirmed: "Zig auto-derefs pointer field access: v.x works for both Vec2 and *const Vec2."

    Note: `return param` where the function returns `T` (not `*const T`) WILL need a deref: `return param.*`. This is a real concern. Check if the codegen for return statements needs adjustment for promoted params. If a function's return type is the struct type (not a pointer), and the returned expression is a promoted parameter, emit `param.*`.

    Similarly, if a promoted param is passed to another function that does NOT have that param promoted, it needs `param.*` to produce a value. This is where Plan 01's `promoted_params` tracking helps — but the actual deref emission is codegen's job.

    For the return case: in the MIR codegen for return statements, check if the returned expression is an identifier that matches a promoted param name. If so, emit `expr.*`.

    For the pass-to-another-function case: if a promoted param (already `*const T`) is passed to a function where that param position is NOT promoted, it needs `expr.*`. But if the target IS promoted, `&expr` would produce `*const *const T` which is wrong. This case should be handled by Plan 01's `promoted_params` tracking — the MIR annotator should NOT apply `value_to_const_ref` to promoted params. But the codegen var-caller `&` emission (Step 4) also needs to skip promoted params.

    SIMPLIFICATION: For phase 8, keep the implementation conservative. The auto-deref covers field access and method calls (the most common pattern for struct params). If bare param usage is uncommon in the codebase, defer explicit deref handling to gap closure. Test with real fixtures to validate.
