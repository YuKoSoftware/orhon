Fix the "file exists in two modules" error that occurs when a multi-file module has a Zig sidecar.

**Root cause investigation**: In Zig 0.15, the same .zig file cannot be used as root_source_file in two different createModule calls. The bug occurs when:
- A module with has_bridges=true gets a `bridge_{name}` module with root_source_file `{name}_bridge.zig`
- AND the same sidecar file is somehow also referenced by another module in the build graph

**Investigation steps for executor**:
1. Read the early sidecar copy in module.zig (~line 537-543) — this copies the sidecar to `{GENERATED_DIR}/{name}_bridge.zig` during module resolution for std:: imports
2. Read the main sidecar copy in main.zig (~line 1102-1133) — this copies with pub fixup
3. Check if both copies produce the same output file, causing a re-registration

**Most likely fix**: The early copy in module.zig (line 537-543) copies the sidecar to `{GENERATED_DIR}/{decl.path}_bridge.zig` BEFORE the main pipeline. This is for stdlib modules. For user modules, the sidecar_path is set but not copied until main.zig. However, the early copy might create a bridge file that later gets picked up when the module is used by multiple targets.

The fix should ensure that in `buildZigContentMulti`:
- When a module has `has_bridges = true`, the sidecar is ONLY referenced via the `bridge_{name}` named module
- The generated module's `.zig` file (codegen output) uses `@import("{name}_bridge")` (named import), never path-based `@import("{name}_bridge.zig")`
- If there are any paths where the sidecar file path ends up as root_source_file for both the bridge module AND another module, deduplicate them

**Specific areas to check and fix in zig_runner.zig**:
1. `buildZigContentMulti` shared module creation (line 1232-1264): If a non-lib module with has_bridges has its .zig file used as mod_{name} root_source_file, and ALSO has bridge_{name} pointing to _bridge.zig — verify these are different files
2. Extra bridge modules (line 1112-1123): check if extra_bridge_modules creates a bridge_{name} that duplicates an already-existing module reference
3. The lib target emission (line 1270-1283): lib_{name} uses {name}.zig as root, bridge_{name} uses {name}_bridge.zig — different files, should be OK
4. Look for any createModule calls that inadvertently point to the same physical file

**Test**: Add a multi-module test to test/07_multimodule.sh that creates a module with multiple .orh files AND a Zig sidecar, verifying it builds without the "file exists in two modules" error. The test should:
- Create module `mylib` with mylib.orh (anchor) + extra.orh (additional file) + mylib.zig (sidecar)
- The sidecar exports a simple function
- The main module imports and uses the bridge function
