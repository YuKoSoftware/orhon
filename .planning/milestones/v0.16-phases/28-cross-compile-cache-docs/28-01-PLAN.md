**BLD-04 — Use-after-free in target flag string (cross-compilation garbled step name):**

In `buildAll()` (around line 77) and `buildWithType()` (around line 153), the `target_flag` string is allocated with `allocPrint` inside an `if` block, then immediately freed by `defer` at the end of that `if` block scope — BEFORE `runZigIn` reads it on line 88/165. This causes the garbled step name.

Fix both locations identically:
1. Declare `var target_flag_alloc: ?[]const u8 = null;` before the `if` block
2. Inside the `if (target.len > 0)` block, allocate and store: `target_flag_alloc = try std.fmt.allocPrint(...)` and append to args
3. Move the `defer` OUTSIDE the if block: `defer if (target_flag_alloc) |tf| self.allocator.free(tf);`
4. This ensures the string lives until function return (after runZigIn completes)

Apply to BOTH functions:
- `buildAll()` lines ~76-80
- `buildWithType()` lines ~152-156

**BLD-05 — -fast cache leak into bin/:**

After `zig build` runs from `.orh-cache/generated/`, both `buildAll()` and `buildWithType()` clean up the `zig-out/` directory inside GENERATED_DIR but do NOT clean up the `.zig-cache/` directory that `zig build` also creates there. While `.zig-cache` inside `.orh-cache/` is hidden, the Zig build system may also create a local `.zig-cache` relative to the project root or inside `bin/` depending on path resolution.

Investigate and fix:
1. After the `deleteTree(generated_zig_out)` call in both functions, also delete `.zig-cache` from GENERATED_DIR:
   ```
   const generated_zig_cache = try std.fs.path.join(self.allocator, &.{ cache.GENERATED_DIR, ".zig-cache" });
   defer self.allocator.free(generated_zig_cache);
   std.fs.cwd().deleteTree(generated_zig_cache) catch {};
   ```
2. Check if `bin/` contains any `.zig-cache` or `zig-cache` directories after a build. If `zig build` creates cache in the project root, also clean that:
   ```
   std.fs.cwd().deleteTree("zig-cache") catch {};
   std.fs.cwd().deleteTree(".zig-cache") catch {};
   ```
3. Apply the cleanup to BOTH `buildAll()` (after line 131) and `buildWithType()` (after line 206)

Do NOT remove zig-cache cleanup from the non-fast path — cleaning up after every build is correct. The cleanup ensures no stale cache leaks regardless of optimization level.
