# Plan: Migrate struct construction from `()` to `{}`

**Date:** 2026-04-05
**Status:** complete
**Scope:** Change struct instantiation syntax from `StructName(field: value)` to `StructName{field: value}` across grammar, builder, resolver, codegen, all `.orh` files, and docs.

## Context

Orhon currently uses `StructName(field: value)` for struct construction, which conflicts with function call syntax (`Foo(x: 1)` is ambiguous between struct init and function call with named arg). This also prevents inline struct arrays like `[Pair{x: 1, y: 2}]` from parsing — the `(` inside `[]` is ambiguous. This is blocking the for-loop tuple capture feature (see `docs/plans/2026-04-05-2100-for-loop-tuple-captures.md`). The `{}` syntax matches Zig, Rust, and Go conventions.

## Approach

Add a new postfix grammar rule `struct_init_access` that uses `{` and `}` instead of `(` and `)`. The builder produces the same `call_expr` AST node with `arg_names` (keeping the existing AST representation). Remove named argument support from `call_access` so that `StructName(field: val)` becomes a parse error. Update all `.orh` source files, test fixtures, and documentation. The codegen already emits Zig `{}` syntax for named-arg calls, so no codegen changes are needed.

## Steps

### 1. Add `struct_init_access` grammar rule
- **Files:** `src/peg/orhon.peg`
- **What:**
  - Add a new rule: `struct_init_access <- '{' _ named_arg_list _ '}'`
  - Add `struct_init_access` to the postfix chain in `postfix_expr`, between `call_access` and `index_access`:
    `postfix_expr <- primary_expr (method_call / field_access / slice_access / index_access / struct_init_access / call_access)*`
  - Note: `struct_init_access` must come AFTER `index_access` (which uses `[`) to avoid ambiguity. It must come BEFORE `call_access` since both are unambiguous (different opener tokens).
  - Actually, ordering between `struct_init_access` and `call_access` doesn't matter since they start with different tokens (`{` vs `(`). Place it after `index_access` for logical grouping.
  - Change `call_access` to use `arg_list` instead of `named_arg_list`:
    `call_access <- '(' _ arg_list _ ')'`
  - This makes `Foo(name: val)` a parse error (the old struct construction syntax).
- **Why:** This is the core grammar change that enables `{}` syntax and removes `()` for struct construction.
- **Difficulty:** medium
- **Isolated:** yes

## Risks & Edge Cases

- **Generic struct construction ambiguity:** `Vec2(T){x: 1, y: 2}` parses as `postfix_expr` where `Vec2(T)` is a `call_access` (generic type instantiation) followed by `{x: 1, y: 2}` as `struct_init_access`. This should work naturally since both are postfix operations chained left-to-right. The codegen already handles this pattern (line 260-264 in `codegen_exprs.zig`).
- **Empty struct construction:** `StructName{}` (zero fields, all defaults) — currently `StructName()` goes through `call_access` with empty args, and codegen emits `StructName{}` (line 249-253). After migration, `StructName{}` should go through `struct_init_access` with empty `named_arg_list`. Verify this works.
- **Cross-module struct construction:** `tester.IsTestType{val: 42}` — the callee is a `field_expr` (module.Type), then `{...}` as `struct_init_access`. This should chain naturally as postfix operations.
- **Block vs struct init:** `foo\n{...}` — the newline token between `foo` and `{` prevents `{` from matching as a postfix. The PEG postfix chain has no `_` between alternatives, so newlines break the chain. This is the desired behavior.
- **Enum variant construction:** `Circle(radius: 5.0)` in the spec uses `()` for data-carrying enum variants. These are NOT yet implemented (see TODO.md). When implemented, they should also use `{}` syntax: `Circle{radius: 5.0}`. Update the spec examples now to use `{}` for consistency, even though the feature isn't implemented.
- **`named_arg_list` reuse:** The `named_arg_list` and `named_or_positional_arg` grammar rules are still needed for `struct_init_access`. They remain in the grammar but are no longer referenced by `call_access`.
- **Compiler-generated AST:** Several passes (resolver, propagation, borrow, ownership, MIR annotator) manually construct `call_expr` nodes with `arg_names = &.{}`. These are for function calls, not struct init, so they are unaffected.

## Out of Scope

- Adding a new `struct_init` AST node kind (the existing `call_expr` with `arg_names` is sufficient for now; a dedicated node could be a future cleanup)
- Named function arguments (not in the spec, `call_access` now only supports positional)
- Data-carrying enum implementation (tracked separately in TODO.md)
- Tuple literal syntax changes (tuples keep `()` syntax)
