  <files>src/peg/grammar.zig</files>
  <action>
1. Add a `labels` field to the `Grammar` struct:
   ```zig
   labels: std.StringHashMapUnmanaged(?[]const u8),
   ```
   This maps rule name -> optional label string. Only rules with `{label: "..."}` get entries.

2. Add a `getLabel` method to `Grammar`:
   ```zig
   pub fn getLabel(self: *const Grammar, rule_name: []const u8) ?[]const u8 {
       if (self.labels.get(rule_name)) |maybe_label| return maybe_label;
       return null;
   }
   ```

3. Add a `labels` field to `GrammarParser`:
   ```zig
   labels: std.StringHashMapUnmanaged(?[]const u8),
   ```
   Initialize as `.{}` in `parseGrammar`.

4. In `GrammarParser.parse()`, after parsing the rule body and storing it (line ~163), add label parsing:
   - After `try self.rule_names.append(...)`, call a new `tryParseLabel` method.
   - `tryParseLabel(name)`: Skip whitespace-only on the current line. If the next chars are `{label:`, parse the quoted string value, skip to `}`, and store in `self.labels.put(self.allocator, name, label_text)`.

5. In `parseGrammar`, pass `parser.labels` into the Grammar return value:
   ```zig
   return Grammar{
       .rules = parser.rules,
       .rule_names = try parser.rule_names.toOwnedSlice(allocator),
       .labels = parser.labels,
       .arena = arena,
   };
   ```

6. The `tryParseLabel` implementation should handle this syntax after the rule body on the same or next line:
   ```
   func_decl
       <- 'func' func_name ...  {label: "function declaration"}
   ```
   Parse: skip whitespace, match `{label:`, skip whitespace, read double-quoted string, skip whitespace, match `}`.

7. Add a test for label parsing:
   ```zig
   test "grammar - parse rule with label" {
       const alloc = std.testing.allocator;
       const src = "foo\n    <- 'bar' baz  {label: \"a foo thing\"}\n";
       var grammar = try parseGrammar(src, alloc);
       defer grammar.deinit();
       try std.testing.expectEqualStrings("a foo thing", grammar.getLabel("foo").?);
   }
   ```

Important: The label syntax uses `{` which could conflict with rule body parsing. Labels appear AFTER the rule body is fully parsed (after parseChoiceExpr returns), so there is no ambiguity. The `{` is NOT a valid start for any PEG expression atom (parseAtom returns null for unknown chars), so parseChoiceExpr will naturally stop before it.
