```zig
fn buildMetadata(ctx: *BuildContext, cap: *const CaptureNode) !*Node {
    const field_pos = cap.start_pos + 1; // after #
    const field = tokenText(ctx, field_pos);
    if (cap.children.len > 0) {
        const value = try buildNode(ctx, &cap.children[0]);
        var extra: ?*Node = null;
        if (cap.children.len > 1) {
            extra = try buildNode(ctx, &cap.children[1]);
        }
        return ctx.newNode(.{ .metadata = .{ .field = field, .value = value, .extra = extra } });
    }
    const dummy = try ctx.newNode(.{ .identifier = field });
    return ctx.newNode(.{ .metadata = .{ .field = field, .value = dummy } });
}
```

Metadata keywords (linkC, cInclude, etc.) are NOT in token_map.zig LITERAL_MAP.
They are matched as contextual identifiers by the PEG engine (checked by text, not token kind).
However, per Phase 22 decision: "every new keyword token must have a string-to-TokenKind entry."
The PEG grammar uses quoted strings like `'cimport'` which match by text — if `cimport` is not
a keyword token, the PEG engine checks the token text against the literal. This works for IDENTIFIER
tokens. Existing metadata keywords (linkC, cInclude, csource, linkCpp) are NOT in LITERAL_MAP and
ARE matched as identifiers by text. Follow the same pattern for `cimport` — do NOT add to LITERAL_MAP
unless existing metadata keywords are there (they are not).

    var cimport_registry = std.StringHashMapUnmanaged([]const u8){};

    for (root_asts) |ast| {
